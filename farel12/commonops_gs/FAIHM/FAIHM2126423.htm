HCM Data Loader Configuration Parameters

The HCM Data Loader configuration parameters control the HCM Data Loader import and load processes. The default settings of the parameters suit most requirements. However, you can change these settings to suit the configuration of your environment and the amount of data being loaded. Users can also override the enterprise settings when loading files either on the Schedule Request page or when using the HCM Data Loader web service.

Setting Configuration Parameters for Performance: Points to Consider

The HCM Data Loader configuration parameters control the HCM Data Loader import and load processes. This topic describes how to set these parameters to optimize multithreaded processing of your files. The examples in this topic provide general guidance. However, you can still load data successfully with parameter values that differ from those suggested here.

Maximum Concurrent Threads for Load

The maximum number of threads that can be used to load a business object depends on the number of servers configured in your environment. Each server supplies 8 threads. Therefore, the maximum number of threads available is 8 multiplied by the number of servers. Business objects are hosted across various servers. Therefore, you must understand not only how many servers are available but also where the business object that you're loading is hosted.

Tip: The Initiate Data Load page in the Data Exchange work area shows the server that hosts each business object.

Load Group Size

The Load Group Size parameter controls the number of business objects that a thread processes at a time. Each concurrent thread processes multiple groups until all objects are processed. By default, the Load Group Size parameter doesn't appear on the Schedule Request page when you submit a file for processing. Instead, at the beginning of the load process, HCM Data Loader calculates the optimal load group size. This calculation is based on the available threads and the number of objects being loaded. You're recommended to leave HCM Data Loader to calculate the load group size. However, you can disable this calculation by setting the Enable Load Group Size Calculation configuration parameter to No. In this case, the Load Group Size parameter appears on the Schedule Request page.

If you disable automatic calculation of the load group size and instead specify a fixed value, then consider these points:

  • The recommended setting for this parameter is 100, which is its default value.

  • Initiating each group of objects incurs an overhead. Therefore, Load Group Size should not be too small.

  • The statistics and status details on the Import and Load Data page are updated only at the end of every group. Therefore, Load Group Size should not be too large for you to monitor progress.

Tip: Change this parameter from the default value only when loading smaller groups of workers. Worker is a particularly complex object with many file lines.

Maximum Concurrent Threads for Import

The Maximum Concurrent Threads for Import parameter controls the number of threads available for the import phase of data loading. Set it to the same value as the Maximum Concurrent Threads for Load parameter. By default, both parameters are set to 8.

Processing Files in Parallel on a Single Server

Regardless of the maximum number of threads you specify when submitting a file, HCM Data Loader can use only available threads. If other activities are using threads, then HCM Data Loader processing must use fewer threads. If you're processing .zip files in parallel for objects hosted on the same server, then consider reducing the maximum concurrent threads value. This approach shares the maximum available threads among your files. Alternatively, supply the business object files in the same .zip file.

Note: When you process in parallel files that contain business objects hosted on separate servers, they each use the maximum threads available.

HCM Data Loader Scope: Explained

The HCM Data Loader Scope parameter determines which business objects you can load using HCM Data Loader and which other data loaders you can use. The HCM Data Loader Scope parameter has two possible values:

HCM Data Loader Scope Value Description

Limited

You can use HCM Data Loader to load only those business objects that aren't supported by its predecessor, HCM File-Based Loader.

Full

You can use HCM Data Loader to load all supported business objects. You can't use HCM File Based Loader, which is disabled.

For customers who upgraded from an earlier release to Release 10, the default value of this parameter is Limited. While you continue to use HCM File-Based Loader, leave HCM Data Loader Scope set to Limited. However, you should plan to use HCM Data Loader in place of HCM File-Based Loader. To ensure complete business-object support when using HCM Data Loader, you set HCM Data Loader Scope to Full.

For new customers, the default value of this parameter has been Full since Release 10.

Note: You can't change HCM Data Loader Scope from Full to Limited.

Using HCM Spreadsheet Data Loader

The version of HCM Spreadsheet Data Loader that you can use depends on the value of the HCM Data Loader Scope configuration parameter. If HCM Data Loader Scope is set to:

  • Full, then you can use HCM Spreadsheet Data Loader to load most objects that HCM Data Loader supports. All HCM Spreadsheet Data Loader Release 12 functions are available.

  • Limited, then you can use the version of HCM Spreadsheet Data Loader that was available in Release 11 and earlier. This version provides a static list of predefined spreadsheets

Note: Spreadsheet data loaders other than HCM Spreadsheet Data Loader are unaffected by the setting of the HCM Data Loader Scope configuration parameter. For example, you can use the payroll batch loader regardless of the value of HCM Data Loader Scope.

Foreign-Object References

When you refer to these objects from objects that you're loading, you use their user keys. (Alternatively, you can use their surrogate IDs, if available.) HCM Data Loader provides business-object documentation for all supported objects. This documentation identifies the user key that you can use for foreign-object references. For example, the position object includes a reference to the business unit, which isn't integration enabled. The position documentation identifies the business-unit name as its user key. Therefore, when loading a position object, you can refer to the associated business unit using the business-unit name. To see the supplied documentation for a business object, click the object name on the Initiate Data Load page.

Reviewing Lists of Values: Explained

The permitted values of many object attributes are defined in lists of values. Some lists of values are predefined and can't be updated in any way. Others contain some values, but you can also add your own. In some cases, you can edit or remove predefined values. In Oracle HCM Cloud, lists of values are defined as lookup types. You're recommended to review the predefined lookup types and make any updates before you attempt to load data that uses them. You may have completed this process during implementation of Oracle HCM Cloud.

Reviewing Lookup Types

To manage lookup types, search for relevant tasks by entering Manage % Lookups in the Setup and Maintenance work area. All available lookup-type tasks are listed. For example, to manage person lookup types, select the Manage Person Lookups task. On the Manage Person Lookups page, select a lookup type to edit. The list includes:

  • Address types

  • Contact relationships

  • Ethnicity

  • Highest education level

  • Honors

  • Marital status

  • Military rank

  • Phone types

  • Religion

Ensure that the defined lookup values support the data that you're planning to load.

Defining the Source-System Owner: Procedure

If you plan to use source keys, then you must add your source-system owner values to the HRC_SOURCE_SYSTEM_OWNER lookup type before you load data. You must have the Application Implementation Consultant or Human Capital Management Application Administrator job role or privileges to perform this task.

Follow these steps:

  1. In the Setup and Maintenance work area, search for and select the Manage Common Lookups task.

  2. On the Manage Common Lookups page, search for the lookup type HRC_SOURCE_SYSTEM_OWNER.

  3. In the search results, select the lookup type to display its lookup codes.

  4. In the Lookup Codes section of the page, click the New icon to add a lookup-code row.

  5. Complete the fields in the row. Ensure that the new code is enabled and the dates are valid for the data that you're loading.

  6. Repeat steps 4 and 5 for additional source-system owner values.

  7. Click Save and Close.

Finding Business-Object Information: Explained

To load a business object or component successfully using HCM Data Loader, you must understand its structure, supported key systems, data formats, and required attributes. You must also understand any business rules that apply to the object. This topic identifies where you can find this information.

Business-Object Details

On the Initiate Data Load page, you can generate data-file templates. When you click the name of a business object in the Business Object column in the Search Results table, you access information about the object. This information includes:

  • The component hierarchy of the object

  • Component details for the currently selected component in the hierarchy. These include:

    • Its parent component, file discriminator, data-file name, and the name of any translation object

      Tip: Click the name of the translation object to see its details.

    • Whether it supports integration keys

    • Whether it's date effective and, if so, whether you can change the component's first start date and last end date

    • The descriptive flexfields that you can upload for the component

  • For each attribute of the component, you can see:

    • Its name, type, and length

    • Whether it's required

    • Whether it's a key value or can be used as a key value and, if so, the key type

    • The names of any alternate user key attributes, which you can use in place of surrogate IDs

Tip: Select Actions - Export to Excel to export the attribute details to a spreadsheet.

Preparing to Load Workers: Points to Consider

Before you load worker objects, you must decide:

  • Whether you want user accounts to be created automatically for those workers.

  • Whether user credentials are to be sent automatically to new users.

  • Which roles are to be provisioned automatically to new users. User accounts created without at least one role are automatically suspended.

This topic describes each of these decisions.

Creating User Accounts Automatically

The enterprise option User Account Creation controls whether user accounts are created automatically for new workers, regardless of how those workers are created. If user accounts are created automatically in your enterprise, then you can prevent accounts being created for individual workers. To prevent account creation, include the GeneratedUserAccountFlag attribute of the user information component and set it to N.

Note: If User Account Creation is set to None, then you can't override it for individual workers.

You can supply a user name in the user information component of the worker object that you upload. Otherwise, user names are in the default format for the enterprise, as specified on the Security Console. The default user-name format is the primary work email. Any changes that you make to the default format apply to all new users in the enterprise, regardless of how they're created.

Sending User Credentials to New Users

If you create user accounts automatically for uploaded workers, then you can notify the users automatically of their user names and passwords. Set the SendCredentialsEmailFlag attribute of the worker object to Y for any worker who is to receive this mail. SendCredentialsEmailFlag is set to N by default.

If you set SendCredentialsEmailFlag to Y for any worker, then you must ensure that a valid notification template is enabled for this event. Two predefined templates exist:

  • The New Account Template is for notifying the user.

  • The New Account Manager Template is for notifying the user's manager.

You can also create custom notification templates. Notification templates are managed on the Security Console. Any changes made to notification templates apply to the enterprise.

Provisioning Roles to New Users

When a user account is created automatically for a worker, roles are provisioned automatically to the user as specified by current role-provisioning rules. Confirm that appropriate role mappings exist for users created by bulk upload.

Note: All user accounts must have at least one role to remain active. If appropriate role mappings don't exist and you're not loading roles for manual assignment, then the new user account is immediately suspended. To avoid this automatic suspension of the account, define role mappings for workers before you load those workers.
