Web Services Overview

A web service is a program that can be accessed remotely using different standards-based languages.

A web service is characterized by three factors:

  • What they do: The business functionality they expose

  • Where they are: The web site that exposes the business functionality

  • How they can be accessed: The set of published interfaces required to use the exposed functionality

Web Service Life Cycle

A business object web service goes through three phases in a life cycle. Oracle Applications Cloud supports public external SOAP services throughout their life cycle, until the services are obsolete.

  1. Active: A service is active when it is delivered the first time, until it is deprecated.

  2. Deprecated: A service is deprecated when it is superseded by a newer version, or if there is a planned obsolescence in a future release. A service in this state is still supported and becomes obsolete in a later release.

    Note: You must use the active version of the service. If you were previously using a service that has been deprecated, then migrate to the new, active version.

  3. Obsolete: A service is obsolete when it is no longer shipped or supported.

Backward Compatibility for Web Services

Web services may be updated as part of an upgrade to a major release or as part of a bundle patch. Any change to a service must be backward compatible and must not break the existing code. Examples of backward compatible changes include:

  • Adding new operations

  • Adding new optional attributes for service data object

Any change to the service that is not backward compatible breaks the existing code that invokes the service. So, backward incompatible changes are not allowed on any public external service. Examples of backward incompatible changes include:

  • Removing an operation

  • Removing a parameter in an operation

  • Changing the parameter type of an existing operation

  • Renaming an attribute of a service data object

  • Adding a mandatory attribute to a service data object

  • Changing the functional behavior of an operation (for example, the service returns a different output when the same values are passed in)

Web Service Versioning

When a backward incompatible change is required for a service, the change is not made directly to the service. Instead, the existing service is left as is, and its life cycle is updated to the Deprecated phase. A new version of the service is introduced, which is effectively a copy of the existing service, and the backward incompatible changes are applied to the copy. The new version of the service is now marked as Active and has its own end point and WSDL. The version number is reflected in the service namespace.
